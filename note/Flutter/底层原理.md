# Flutter 底层原理

> 一句话：**Widget 是配置单，Element 是「挂载点」，RenderObject 才是真正干活的——约束下行、尺寸上行，再布局、绘制、合成。**

---

## 三层树：谁在干什么

Flutter 的 UI 不是一棵树搞定，而是三棵互相关联的树协同工作：

| 树 | 角色 | 一句话 |
|----|------|--------|
| **Widget 树** | 声明式配置 | 开发者写的「说明书」，不可变，可随时被替换 |
| **Element 树** | 中间层 | 把 Widget 挂到渲染体系上，持有 State、管理更新 |
| **RenderObject 树** | 渲染层 | 算尺寸、算位置、画像素，和 GPU 打交道 |

- **Widget**：轻量、可大量创建；递归组合，最终由 `RenderObjectWidget` 造出 RenderObject。
- **Element**：Widget 的「实例化」——对应关系稳定，Widget 换了他负责对比、更新子节点。
- **RenderObject**：存几何信息（大小、位置），负责 layout、paint、compositing；常见 2D 用的是 `RenderBox`（min/max 宽高约束）。

---

## 渲染管线（一帧里发生什么）

每帧大致顺序：

1. **Layout（布局）**：约束从父往下传，子把尺寸往上传；单次遍历，每个 RenderObject 最多被访问两遍（下、上），布局更新可做到亚线性。
2. **Paint（绘制）**：按几何信息生成绘制指令（Layer/Picture）。
3. **Compositing（合成）**：合成场景，交给引擎光栅化上屏。

一句话：**约束向下，尺寸向上；布局完再画，画完再合成。**

---

## Layer 与合成（Compositing）

- **Layer**：绘制结果的抽象，可被 GPU 独立光栅化；多个 Layer 按 z-order 合成一张图。
- **为什么要 Layer**：避免整屏重绘——只有「脏」的 Layer 才重画，其余复用缓存，减少 Overdraw、提升帧率。
- **常见 Layer**：`OffsetLayer`、`PictureLayer`（存 Picture）、`TransformLayer` 等；RepaintBoundary 会触发新 Layer 的生成。
- **一句话**：Layer 是「可复用的绘制块」，合成阶段把它们叠在一起上屏。

---

## Key 与 Element 复用

- **Key**：在 Widget 树中标识「同一个逻辑节点」，帮助 Element 在兄弟顺序变化时决定是复用还是新建。
- **LocalKey**：同一父下唯一即可，如 `ValueKey`、`ObjectKey`；**GlobalKey**：全局唯一，可跨父节点、拿 State 或 Find 当前 Widget。
- **复用逻辑**：同一层级、同类型 Widget + 同 Key → Element 复用、只更新；否则可能销毁旧 Element、建新的，State 就丢了。
- **一句话**：Key 告诉 Flutter「这是不是同一个东西」——列表重排、动态表单里用对 Key 能避免状态错乱或多余重建。

---

## InheritedWidget 与依赖注入

- **InheritedWidget**：把数据「挂在」Element 树上，子孙通过 `context.dependOnInheritedWidgetOfExactType<T>()` 或 `context.inheritFromWidgetOfExactType` 取用。
- **依赖关系**：当 InheritedWidget 更新时，依赖过它的 Element 会标记 dirty，下一帧重建；没依赖的不会动。
- **Provider/Riverpod 底层**：Provider 就是 InheritedWidget + 状态封装；Riverpod 不依赖 BuildContext，用 Provider 容器 + Ref 取数据。
- **一句话**：InheritedWidget 是「树上的广播站」——谁 depend 谁就会在它变的时候收到更新。

---

## BuildOwner 与 PipelineOwner

- **BuildOwner**：管「什么时候、哪些 Element 需要 build」；持有 dirty 列表，驱动 build 阶段（只重建标记过的 Element）。
- **PipelineOwner**：管「哪些 RenderObject 需要 layout/paint」；持有 layout/paint 的 dirty 列表，驱动渲染管线。
- **关系**：BuildOwner 管「配置层」的更新，PipelineOwner 管「渲染层」的更新；一帧里先处理 build，再 layout → paint → composite。
- **一句话**：两个 Owner 分别管「谁要重建」和「谁要重画」，是帧调度的核心协调者。

---

## 约束传递与布局边界

- **约束（Constraints）**：父给子的「尺寸范围」，如 BoxConstraints(min/max width/height)；子必须在约束内选一个 size 回报给父。
- **传递方向**：约束自上而下，尺寸自下而上；父一次 layout 子，子返回 size 后父才能确定自己的 layout。
- **布局边界**：某些 RenderObject（如 `RenderView`、带 `layoutBoundary` 的）会打断约束传递，用于做「相对布局」或优化。
- **一句话**：布局就是「父问子：给你这么大空间，你要多大？」子答完，父再算自己的位置和剩余空间。

---

## RepaintBoundary 原理

- **RepaintBoundary**：在 RenderObject 树里插一个「重绘边界」，其子树会画到独立 Layer 上，重绘时只重画这个 Layer，不影响兄弟。
- **何时用**：动画多的子树、复杂静态子树（不想被父的 repaint 牵连）、列表里每项可独立重绘时。
- **代价**：多一层 Layer，合成与内存略增；乱用会适得其反。
- **一句话**：RepaintBoundary = 「把这棵子树单独画一块」，脏了只重画这块，省 GPU。

---

## 帧调度与 VSync

- **VSync**：垂直同步信号，显示器刷新一次发一次；Flutter 引擎按 VSync 节奏调度帧，避免撕裂、控制 60/120fps。
- **一帧流程**：VSync 到来 → 动画/构建/布局/绘制/合成 → 提交给 GPU → 等下一个 VSync。
- **Jank**：一帧超过 16.6ms（60fps）就会掉帧；原因常是 build/layout/paint 过重或主 isolate 被阻塞。
- **一句话**：Flutter 跟着屏幕刷新节奏走，一帧内做完所有事才能不卡。

---

## Flutter 与 Skia / Impeller

- **Skia**：Flutter 长期使用的 2D 渲染引擎，跨平台、成熟；在 iOS/Android 上把 Layer 光栅化成 GPU 纹理。
- **Impeller**：Flutter 新一代渲染引擎（iOS/Android 逐步启用），预编译 shader、减少 jank，目标更稳定的 60fps。
- **关系**：应用层写的都是 Flutter API，底层换 Skia/Impeller 对业务透明；关注官方迁移说明即可。
- **一句话**：Skia 是现在的「画笔」，Impeller 是下一代「画笔」，Flutter 正在逐步切换。

---

## Dart 与 Flutter 的关系

- Flutter 用 **Dart** 写，Dart 跑在 **Dart VM**（调试）或 **AOT 编译**（发布）里。
- **Isolate**：Dart 的「线程」模型，每个 isolate 有独立堆和 **Event Loop**（microtask 优先于 event queue）。
- **热重载**：把改动的 Dart 代码注入 VM，保留应用状态，只重建受影响子树，比整包重启快得多。

---

## 三棵树协作流程（从 setState 到上屏）

1. **setState()**：标记当前 State 对应 Element 为 dirty，并通知 BuildOwner。
2. **BuildOwner.flushBuildScope**：对 dirty Element 执行 build，用新 Widget 替换旧引用，递归更新子节点（可复用 Element 或新建）。
3. **PipelineOwner**：被标记需要 layout 的 RenderObject 做 layout；需要 paint 的做 paint，产出 Layer。
4. **合成与上屏**：Layer 树交给引擎（Skia/Impeller）光栅化，提交给 GPU，等 VSync 显示。

一句话：**setState → 标记 dirty → build → layout → paint → composite → 显示。**

---

## 为什么 Widget 要设计成不可变（immutable）？

- **复用与对比**：Widget 轻量、可大量创建；Element 通过对比「新旧 Widget」决定是更新、替换还是移除子节点，不变性保证对比简单、可预测。
- **声明式**：每次 build 返回的是「当前状态下的完整描述」，框架负责把描述变成真实 UI，而不是开发者手改某一块——减少状态不同步的 bug。
- **一句话**：不可变 = 每次都是新配置单，框架只负责「按新单子干活」，不记旧单子。

---

## 布局的「双向传递」具体是什么？

- **下传约束**：父调用 `child.layout(constraints)`，把 BoxConstraints 传给子；子必须在这个范围内选一个 Size。
- **上传尺寸**：子 `layout` 里确定自己的 size，父用 `parentData` 或后续 layout 决定子在自己坐标系里的位置（offset）。
- **多子布局**：父一次 layout 所有子，根据子的 size 计算剩余空间和位置（如 Row/Column 的 Flex、Stack 的 positioned）。
- **一句话**：父说「你最多这么大」，子说「我要这么大」，父再根据所有子的答案排位置。

---

## RenderBox 的约束类型

- **紧约束（Tight）**：min == max，子只能选这一个尺寸，如 `BoxConstraints.tight(Size(100,50))`。
- **松约束（Loose）**：子可在 0～max 之间选，如 `BoxConstraints.loose(Size(100,100))` 表示最大 100×100。
- **无界（Unbounded）**：某一轴 max 为 infinity，如 ListView 在 ScrollView 里时竖轴无界；子必须能处理无界（如 ListView 自己滚动）。
- **一句话**：约束越「紧」子越没选择，越「松」子越自由，无界要小心溢出。

---

## Picture 与 Layer 的关系

- **Picture**：记录一系列绘制指令（线、圆、文字等），可重复播放；Paint 阶段 RenderObject 产出 Picture。
- **PictureLayer**：挂一个 Picture，是 Layer 的一种；合成时把多个 Layer 的 Picture 按 z-order 叠在一起光栅化。
- **一句话**：Picture 是「画了什么」，Layer 是「这一块画在哪一层」；合成阶段按层叠加上屏。

---

## 小结：一帧里各阶段谁在忙

| 阶段 | 负责方 | 产出/作用 |
|------|--------|-----------|
| Build | BuildOwner + Element | 用新 Widget 更新 Element 树、标记需 layout 的 RenderObject |
| Layout | PipelineOwner + RenderObject | 约束下行、尺寸上行，得到几何信息 |
| Paint | RenderObject | 产出 Picture、挂到 Layer |
| Composite | Layer 树 | 按 z-order 合成，交给引擎光栅化 |
| Rasterize | Skia/Impeller | GPU 执行，上屏 |

一句话：**Build 更新「配置」，Layout 算「位置大小」，Paint 画「像素」，Composite 叠「层」，最后交给 GPU。**

## 常见坑与排查

- **setState 时机**：在 `build` 里调 `setState` 会触发「build → setState → 再 build」的循环或重复重建；setState 应只在事件、回调、Timer、Future 完成等**非 build 路径**里调用。若必须在「数据到了再更新」，用 `WidgetsBinding.instance.addPostFrameCallback` 延到本帧结束后再 setState。
- **build 里创建对象**：每次 build 都会执行，在 build 里 `new List()`、`new Controller()` 等会导致每帧新建对象、子组件拿不到稳定引用、或重复注册监听。列表/控制器应放在 State 的成员变量、`initState` 里初始化，或通过状态管理注入。
- **列表不用 builder**：`ListView(children: list.map(...).toList())` 会一次性 build 全部子项，长列表必卡。改用 `ListView.builder`（懒加载，只 build 可见项），或 `ListView.separated` 需要分隔时用。
- **无 Key 导致列表状态错乱**：列表项顺序会变（增删、排序）时，若不用 Key 或 Key 不稳定，Element 可能复用到错误的 State（如输入框内容串行）。给列表项加稳定的 `ValueKey(id)` 或 `ObjectKey(item)`，让 Flutter 正确复用对应 Element。
- **一句话**：setState 别在 build 里、大对象别在 build 里建、长列表用 builder、会重排的列表项给好 Key。
