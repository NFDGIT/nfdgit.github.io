# Flutter 高阶用法

状态管理、性能、平台通道与工程化要点，言简意赅。

---

## 一、状态管理选型

| 方案 | 特点 | 适合场景 |
|------|------|----------|
| **Provider** | 基于 InheritedWidget，上手快、样板少 | 小中项目、快速迭代 |
| **Riverpod** | 不依赖 Context、编译期安全、AsyncValue 友好 | 中大型、新项目首选 |
| **Bloc** | 事件→状态、逻辑与 UI 彻底分离，样板多 | 复杂流程、企业级 |

- **一句话**：小项目用 Provider 够用；要可扩展、可测试，选 Riverpod；强流程规范选 Bloc。
- 避免在根用 Provider 塞大对象导致整树重建；Riverpod 的细粒度依赖能减少无效 rebuild。

---

## 二、性能优化

- **列表**：用 `ListView.builder` / `GridView.builder`，懒加载子项。
- **const**：能标 const 的 Widget 尽量标，利于复用、减重建。
- **拆 Widget**：大树拆成小 Widget，便于 Flutter 做 diff 与局部更新。
- **RepaintBoundary**：隔离重绘区域，复杂或动画多的子树可包一层。
- **避免在 build 里做重计算**：放到 `initState`、Future、或状态管理侧。

---

## 三、平台通道（Platform Channel）

- **用途**：Dart 与原生（iOS/Android）互调——用原生 API、写插件、拿设备能力。
- **常见类型**：MethodChannel（调用方法）、EventChannel（原生推事件到 Dart）。
- **注意**：跨通道是异步的，主线程不阻塞；大数据考虑分片或换更高效的通道方式。

---

## 四、混合开发与插件

- **混合开发**：在现有 iOS/Android 工程里嵌 Flutter 模块（Flutter Engine + 业务 Dart），通过路由与通道和原生交互。
- **插件（Plugin）**：封装平台能力给 Dart 用，一般含 Dart API + 各平台实现；发布到 pub.dev 供他人依赖。
- **包（Package）**：纯 Dart 可只发 Dart 包；带原生代码的用 Plugin 模板。

---

## 五、小结

- 状态管理：按项目规模与团队习惯选 Provider / Riverpod / Bloc，说清数据流。
- 性能：builder 懒加载、const、拆 Widget、RepaintBoundary、少在 build 里算。
- 平台与工程：Channel 做桥接；混合开发与插件搞清楚 Engine、入口与通信方式。
