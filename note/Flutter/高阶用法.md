# Flutter 高阶用法

状态管理、性能、平台通道与工程化要点，言简意赅。

---

## 一、状态管理选型

| 方案 | 特点 | 适合场景 |
|------|------|----------|
| **Provider** | 基于 InheritedWidget，上手快、样板少 | 小中项目、快速迭代 |
| **Riverpod** | 不依赖 Context、编译期安全、AsyncValue 友好 | 中大型、新项目首选 |
| **Bloc** | 事件→状态、逻辑与 UI 彻底分离，样板多 | 复杂流程、企业级 |

- **一句话**：小项目用 Provider 够用；要可扩展、可测试，选 Riverpod；强流程规范选 Bloc。
- 避免在根用 Provider 塞大对象导致整树重建；Riverpod 的细粒度依赖能减少无效 rebuild。

### Provider 简例

```dart
// 定义
ChangeNotifierProvider(create: (_) => MyModel()),

// 使用
context.watch<MyModel>();  // 依赖，会随变化重建
context.read<MyModel>();   // 只读一次，不监听
```

### Riverpod 核心概念

- **Provider**：声明式依赖，不依赖 BuildContext；`ref.watch(xxxProvider)` 监听，`ref.read` 单次读。
- **AsyncValue**：`loading` / `data` / `error` 三态，配合 FutureProvider/StreamProvider 用。
- **ProviderScope**：包在 app 根，容器；可多 Scope 做测试或隔离。

### Bloc 核心概念

- **Bloc/Cubit**：接收 Event（或直接方法），emit 新 State；UI 用 BlocBuilder/BlocListener 消费。
- **单向数据流**：Event → Bloc → State → UI；易测试、可回溯。
- **选型**：流程多、状态机明显时用 Bloc；简单递增状态可用 Cubit。

## 二、性能优化

### 清单速查

- **列表**：用 `ListView.builder` / `GridView.builder`，懒加载子项。
- **const**：能标 const 的 Widget 尽量标，利于复用、减重建。
- **拆 Widget**：大树拆成小 Widget，便于 Flutter 做 diff 与局部更新。
- **RepaintBoundary**：隔离重绘区域，复杂或动画多的子树可包一层。
- **避免在 build 里做重计算**：放到 `initState`、Future、或状态管理侧。
- **图片**：用 `cached_network_image`、合适分辨率；大图先缩略再展示。
- **避免在 build 里创建 Future/Stream**：会每帧新建，应用 `FutureBuilder`/`StreamBuilder` 时把 Future/Stream 提到外层或状态里。

### 常见坑

- **在 build 里 setState**：会死循环或重复 build；setState 应来自事件、回调、Timer 等。
- **列表不用 builder**：`ListView(children: list)` 会一次性 build 全部，长列表必卡；改用 `ListView.builder`。
- **过度使用 RepaintBoundary**：每层都包反而多 Layer、合成更贵；只包「确实独立重绘」的子树。

### RepaintBoundary 何时用

- 动画多的子树（如每项一个动画的列表）、复杂但静态的卡片（不想被父的 repaint 牵连）、自定义 painter 的复杂区域。用 DevTools 的「Show repaint bounds」看边界，再决定包哪一层。

### const 详解

- const 构造函数在编译期求值，同一 const 实例全局复用；Flutter 在 Element 对比时若新旧都是 const 且相等会直接复用。列表 item、装饰性 Widget 能 const 就 const。

## 三、平台通道（Platform Channel）

### 用途

- Dart 与原生（iOS/Android）互调：调原生 API、写插件、拿设备能力（相机、蓝牙、文件等）。

### 常见类型

| 类型 | 用途 |
|------|------|
| **MethodChannel** | 双向调用：Dart 调原生方法、原生返回结果 |
| **EventChannel** | 原生持续推事件到 Dart（如传感器、流） |
| **BasicMessageChannel** | 自定义编解码的二进制/消息 |

### MethodChannel 简例

```dart
// Dart
static const channel = MethodChannel('samples.flutter.dev/battery');
final int level = await channel.invokeMethod<int>('getBatteryLevel');
```

```kotlin
// Android
MethodChannel(flutterEngine.dartExecutor.binaryMessenger, "samples.flutter.dev/battery")
  .setMethodCallHandler { call, result ->
    if (call.method == "getBatteryLevel") { ... result.success(level) }
  }
```

### 注意点

- 跨通道是异步的，主线程不阻塞；大数据考虑分片或换更高效的通道方式。异常要在原生侧捕获并通过 result.error 回传，避免崩溃。

## 四、混合开发与插件

### 混合开发步骤简述

1. 现有工程添加 Flutter 模块（或创建 Flutter module）。
2. 集成 Flutter Engine：Android 用 `FlutterEngine` + `FlutterEngineCache`，iOS 用 `FlutterEngine` + 挂到 VC。
3. 打开 Flutter 页面时启动/复用 Engine，通过 MethodChannel 与原生通信；退出时视情况销毁或缓存 Engine。
4. 路由与深链接：由原生决定何时打开 Flutter 页、传参通过 Channel 或 Route 参数。

### 插件（Plugin）开发要点

- **创建**：`flutter create --template=plugin xxx`，得到 dart 层 + android/ios 实现目录。
- **Dart API**：在 lib 里暴露接口，内部用 MethodChannel/EventChannel 调平台。
- **各平台实现**：android 下 Kotlin/Java，ios 下 Swift/ObjC，实现 channel 的 handler。
- **发布**：`flutter pub publish` 前检查 pubspec、LICENSE、CHANGELOG；版本遵守语义化。

### 包（Package）与插件区别

- **Package**：可纯 Dart，只做逻辑或 UI 组件，无平台代码。
- **Plugin**：含 Dart + 平台代码，封装设备或系统能力；发布后别人 `dependencies` 即可用。

## 五、测试与 CI

### 单元测试

- 测业务逻辑、工具函数、状态转换；不依赖 Flutter 框架时用 `test` 包即可。mock 可用 `mockito`。

### Widget 测试

- `testWidgets` + `tester.pumpWidget` 挂 Widget；`find.byType` / `find.text` 找子组件；`tester.tap`、`tester.pump` 模拟交互与帧推进；断言 `expect(find.text('x'), findsOneWidget)`。

### 集成测试

- `integration_test` 包，跑在真机/模拟器上，测完整流程；可配合 Driver 或直接 run。

### CI 常见流水线要点

- 安装 Flutter、缓存 pub cache；`flutter pub get`、`flutter analyze`、`flutter test`；必要时 `flutter build apk/ios` 做构建验证。多环境可用 matrix 或多 job 区分 channel（stable/beta）。

## 六、小结

- **状态管理**：按项目规模与团队习惯选 Provider / Riverpod / Bloc，说清数据流。
- **性能**：builder 懒加载、const、拆 Widget、RepaintBoundary、少在 build 里算；避免 build 里创建 Future/ setState。
- **平台与工程**：Channel 做桥接；混合开发与插件搞清楚 Engine、入口与通信方式；测试分单元/Widget/集成，CI 跑 analyze + test。

### 状态管理选型表（细化）

| 维度 | Provider | Riverpod | Bloc |
|------|----------|----------|------|
| 依赖 Context | 是 | 否 | 否 |
| 异步支持 | 需自己封装 | AsyncValue 内置 | Stream 友好 |
| 测试 | 需包 Provider | 易 mock、多 Scope | 易测 Event→State |
| 学习成本 | 低 | 中 | 中高 |
| 样板量 | 少 | 中 | 多 |

### EventChannel 简例

- Dart 侧：`EventChannel('name').receiveBroadcastStream()` 得到 Stream；listen 收事件。
- 原生侧：实现 StreamHandler，在 onListen 里开始推数据，cancel 时停止。适合传感器、蓝牙、持续回调等。

### 混合开发通信注意

- 约定好 channel name、方法名、参数与返回的编解码（JSON 或标准类型）；异常统一用 result.error 回传；避免在原生主线程做重活，可切子线程再 result.success。

### 插件发布前检查

- pubspec 的 version、description、homepage；LICENSE 文件；CHANGELOG 更新；`flutter pub publish --dry-run` 自检；确认各平台示例能跑通。

### 性能分析流程简述

1. 用 Profile 模式跑（非 Debug）；打开 DevTools Performance。
2. 录制几秒操作，看帧时间柱状图，找出超过 16.6ms 的帧。
3. 点进该帧看 CPU 火焰图，定位是 build/layout/paint 哪一段耗时。
4. 结合 Widget 树与 RepaintBoundary 边界，决定拆 Widget、加 const、或加 RepaintBoundary。

### 状态管理「依赖」与「重建」关系

- Provider watch：依赖该 Provider，其 notifyListeners 时该 Widget 会重建。
- Riverpod ref.watch：依赖该 provider，值变时对应 Widget 重建；ref.read 不建立依赖。
- 避免在根放「大而全」的 Provider，导致一点变化整树重建；按功能拆成多个 Provider 或改用 Riverpod 细粒度。

### 混合开发中 Flutter 引擎缓存

- 多页用 Flutter 时，可预创建多个 FlutterEngine 并缓存，打开 Flutter 页时直接取，避免每次创建引擎的冷启动；退出页时可选择销毁或放回缓存，视内存与体验权衡。

### 单元测试示例（逻辑）

```dart
test('counter increments', () {
  final cubit = CounterCubit();
  expect(cubit.state, 0);
  cubit.increment();
  expect(cubit.state, 1);
});
```

### Widget 测试示例

```dart
testWidgets('tap button', (tester) async {
  await tester.pumpWidget(MyApp());
  await tester.tap(find.byIcon(Icons.add));
  await tester.pump();
  expect(find.text('1'), findsOneWidget);
});
```

### CI 里 Flutter 安装与缓存（示例思路）

- 用官方 action 或自写：安装指定 channel/version 的 Flutter，把 `~/.pub-cache` 和 `flutter sdk` 做 cache key，加速后续 job。

### GetX / Get 简要对比

- GetX：除状态管理外带路由、依赖注入、国际化等；写法简短，但「魔法」多、与 Flutter 惯用模式差异大。选型时权衡团队习惯与长期维护；新项目更常见选 Riverpod/Bloc。

### 多 Flutter Engine 场景

- 部分场景（如多 Tab 各一个 Flutter 页、或多窗口）会多 Engine；每个 Engine 独立 isolate、独立内存；注意资源与 channel 不要混用，每个 Engine 有自己的 channel 实例。

### 调试 Release 包崩溃

- 保留 `--split-debug-info` 产出的符号；崩溃堆栈用 `symbolize` 或上传到符号服务器还原；可配合 Sentry 等做上报与符号关联。

### 资源与字体优化

- 图片放合适分辨率、用 `flutter pub run flutter_native_splash` 等工具生成多分辨率；字体只包含用到的字重与子集，可减体积。

### 路由与深链接

- 声明式路由（Navigator 2.0 / go_router）用 path 表示栈；深链接进来时解析 path、恢复栈状态；需在原生侧配置 scheme 与 host，Flutter 侧用 `WidgetsBinding.instance.platformDispatcher.defaultRouteName` 或 go_router 的 deep link 配置接收。

### 错误边界与全局捕获

- FlutterError.onError 捕获 Flutter 框架错误；runZonedGuarded 包住 runApp 可捕 isolate 内未捕获异常；Release 下可上报到 Sentry 等，便于排查线上问题。

### 状态恢复（Restoration）

- 用 RestorationMixin、RestorationBucket 保存/恢复页面状态；应用被系统回收再打开时恢复滚动位置、表单等；需在 MaterialApp 上设 restorationScopeId。
