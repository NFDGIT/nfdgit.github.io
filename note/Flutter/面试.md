# Flutter 面试题速览

按考点分类，要点为主，便于复习。以官方文档与最新实践为准。

---

## 一、基础与概念

**StatelessWidget vs StatefulWidget**

- **StatelessWidget**：无状态、不可变，适合静态或由父组件驱动的 UI。
- **StatefulWidget**：带 `State`，通过 `setState()` 触发重建，用于会变的 UI。

**BuildContext 是什么？**

- 表示 Widget 在 **Element 树**里的「位置」；用来拿 Theme、Navigator、找父/子、Overlay 等，不是「当前 Widget 的引用」。

**const 构造函数**

- 编译期常量，减少重复创建、利于 Flutter 复用 Element，能加就加。

---

## 二、原理类

**Widget / Element / RenderObject 三者关系？**

- Widget：配置单（轻量、可大量建）。Element：挂载点，对应 Widget 实例、管更新。RenderObject：真正做 layout/paint 的，存几何信息。见 [底层原理](底层原理.md)。

**Event Loop 与 Isolate**

- Dart 单线程 + Event Loop；microtask 先于 event queue。Isolate 之间内存隔离，靠消息通信。

**热重载（Hot Reload）原理**

- 把改动的 Dart 代码注入 VM，保留应用状态，只重建受影响子树，比完整重启快。

## 三、状态管理

| 方案 | 特点 | 适用 |
|------|------|------|
| **Provider** | 基于 InheritedWidget，简单、样板少 | 中小项目、入门 |
| **Riverpod** | 不依赖 BuildContext、编译期安全、异步友好 | 中大型、推荐主流 |
| **Bloc** | 事件→状态、逻辑与 UI 解耦，样板多 | 复杂业务、企业级 |

面试常问：为什么用状态管理？数据流怎么走？选型依据？（可答：项目规模、团队习惯、是否需要强约束与可测试性。）

---

## 四、性能与实践

- **ListView** 用 `ListView.builder`，按需构建，不一次性建全部子项。
- 大 Widget 树拆成小 Widget，利于复用与局部重建。
- **Platform Channel**：Dart ↔ 原生（iOS/Android）通信桥，用于调原生 API、插件。
- **MaterialApp vs WidgetsApp**：MaterialApp 自带 Material 设计、主题、路由等；WidgetsApp 更底层、更少默认行为。

---

## 五、小结

核心考察：**数据流、响应式更新、三棵树、状态管理选型、性能意识**。把「谁驱动谁、何时重建」说清楚，比背题更管用。
