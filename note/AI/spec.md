# 规范式编程（Specification Programming）详解

> 生动形象、系统全面地介绍规范式编程：从定义到实践，从理论到框架。

---

## 一、定义

**规范式编程**（Specification Programming / Spec-Driven Development，简称 SDD）是一种**以规范为核心、先定规矩再写代码**的编程方法论。

### 核心思想

> **在动键盘之前，先把「要做什么」说清楚。**

就像盖房子：传统编程是拿起砖就砌，规范式编程是先画好图纸、定好尺寸、列好材料清单，再按图施工。规范就是那张「图纸」——它可以是：

- **类型约束**：用类型系统表达「这个函数只接受非空字符串」
- **测试用例**：用测试描述「输入 A 必须得到 B」
- **契约**：前置条件、后置条件、不变式（Design by Contract）
- **形式化规约**：TLA⁺、Alloy、Z 语言等数学化描述
- **自然语言文档**：需求说明、设计文档、任务清单

### 一句话概括

**规范式编程 = 把「需求」和「约束」写清楚，再让代码成为这些规范的实现。**

---

## 二、诞生的背景与解决的问题

### 传统开发的痛点

| 问题 | 表现 |
|------|------|
| 需求漂移 | 写着写着就偏离了最初目标 |
| 文档滞后 | 代码改了，文档没改，最后没人信文档 |
| 重构恐惧 | 不敢动老代码，怕改坏 |
| 沟通成本 | 产品、测试、开发各说各话 |
| AI 随意发挥 | 大模型生成代码时「脑补」需求、遗漏边界 |

### 「Vibe Coding」的困境

在 AI 辅助编程时代，很多人习惯**一句话 prompt 直接要代码**，俗称 Vibe Coding（氛围感编程）。问题在于：

- AI 不知道你的项目结构
- AI 容易遗漏边界条件
- AI 会「自由发挥」，偏离你的真实需求
- 上下文一长就「失忆」

### 规范式编程的应对

**把规范写下来，让 AI 按规范执行。**

规范成为：

1. **需求锚点**：明确「要做什么」
2. **实现约束**：明确「不能做什么」「必须怎么做」
3. **验收标准**：测试、检查清单
4. **活文档**：规范即文档，代码即规范的实现

## 三、有无规范式编程的对比

### 场景：开发一个用户登录功能

#### 无规范式编程（传统 / Vibe Coding）

```
开发者：帮我写个登录功能
AI：好的（生成一堆代码，可能用了你不想要的库、不符合项目结构、漏了验证逻辑）
开发者：不对，我要的是 xxx...
AI：好的（又改一版，可能引入新问题）
... 反复拉扯
```

**结果**：代码能跑，但和预期有偏差，文档缺失，后续维护靠「猜」。

#### 有规范式编程

```
1. 先写规范（proposal / spec）：
   - 登录方式：邮箱 + 密码
   - 校验：密码至少 8 位，含数字和字母
   - 错误处理：3 次失败锁定 15 分钟
   - 安全：密码加盐存储，不记录明文
   - 与现有 auth 模块集成，复用 session 逻辑

2. 人工审核规范，确认无误

3. AI 按规范逐项实现

4. 完成后，规范归档为项目文档
```

**结果**：需求清晰、实现可控、文档自动维护、后续改动有据可查。

### 对比小结

| 维度 | 无规范 | 有规范 |
|------|--------|--------|
| 需求理解 | 靠口头/零散沟通 | 规范即共识 |
| 代码质量 | 易偏离、易遗漏 | 有约束、可验证 |
| 文档 | 易过时、易缺失 | 规范即活文档 |
| 重构 | 不敢动 | 规范不变即可放心改 |
| AI 协作 | 易「乱写」 | 有契约，输出可控 |

## 四、主流框架与优缺点

### 1. OpenSpec（Fission-AI）

轻量 Spec-Driven Development 框架，人与 AI 先对齐规范再写代码。每个变更有独立目录（proposal、specs、design、tasks），支持 20+ AI 工具，规范与代码同仓持久化。

**安装与使用**：`npm install -g @fission-ai/openspec@latest` → `openspec init` → 对 AI 说 `/opsx:new <功能名>` 开始。需 Node.js 20.19.0+。

**完整文档**：[openspec.pro](https://openspec.pro/) · [GitHub](https://github.com/Fission-AI/OpenSpec)

---

### 2. GitHub Spec Kit

**定位**：GitHub 官方开源工具包，支持 Spec-Driven Development。

**四阶段工作流**：

1. **Specify**：定义要做什么、解决什么问题、成功标准（偏业务，不写技术细节）
2. **Plan**：加入技术约束（技术栈、架构、合规要求）
3. **Tasks**：拆成可评审、可测试的小任务
4. **Implement**：按任务顺序实现

**优点**：

- 官方背书，生态成熟
- 支持 20+ AI 工具（Cursor、Claude、Copilot、Gemini 等）
- 适合从 0 到 1、探索性开发、老项目改造
- 规范作为「可执行制品」，直接驱动代码生成

**缺点**：

- 流程相对重，小项目可能显得繁琐
- 需要一定学习成本

**使用**：通过 `/speckit.specify`、`/speckit.plan`、`/speckit.tasks` 等斜杠命令与 AI 交互。

---

### 3. Cursor Plan 模式

**定位**：Cursor 内置的「先规划后执行」模式，本质是规范式工作流的一种实现。

**是否算规范式编程？**  
**是的。** Plan 模式就是「先写规范（计划），再按规范执行」——计划即规范的一种形式。

**工作流程**：

1. **调研**：AI 扫描代码库，收集相关文件
2. **澄清**：向开发者提问，确保理解需求
3. **生成计划**：输出 Markdown 格式的详细计划（文件路径、代码引用、步骤）
4. **审阅编辑**：开发者可修改计划
5. **执行**：点击 Build，AI 按计划逐步实现

**触发方式**：在 Agent 输入框按 **Shift + Tab**，或输入复杂任务时 Cursor 自动建议。

**优点**：

- 无需额外安装，Cursor 自带
- 计划可保存为 Markdown，便于团队共享
- 适合多文件、多模块、重构、新功能开发
- 减少 AI 幻觉和逻辑跳跃

**缺点**：

- 依赖 Cursor，无法在其他 IDE 复用
- 计划格式相对自由，不如 OpenSpec / Spec Kit 结构化

---

### 4. 形式化方法工具（TLA⁺、Dafny、Alloy）

**定位**：用数学/形式化语言描述系统行为，并做自动验证。

| 工具 | 适用场景 | 特点 |
|------|----------|------|
| TLA⁺ | 分布式系统、并发 | 亚马逊用于验证 DynamoDB |
| Dafny | 程序正确性证明 | 微软用于安全关键组件 |
| Alloy | 建模与约束分析 | 轻量形式化建模 |

**优点**：严谨、可自动验证、能发现深层逻辑错误。

**缺点**：学习曲线陡、开发成本高，适合关键系统，不适合日常业务开发。

## 五、老项目适合什么框架？怎么用？

### 老项目的特点

- 已有代码，结构可能混乱
- 文档缺失或过时
- 不敢大改，怕牵一发动全身
- 需要逐步加功能、做重构

### 推荐：**OpenSpec** 或 **Cursor Plan 模式**

| 场景 | 推荐 |
|------|------|
| 只用 Cursor，不想装额外工具 | Cursor Plan 模式 |
| 希望规范可版本管理、团队共享 | OpenSpec |
| 需要更结构化、可复用的流程 | Spec Kit |

### 若二选一：已有项目更选 OpenSpec

若在 OpenSpec 与 Spec Kit 中**必须选一个**用于已有项目，更推荐 **OpenSpec**。主要原因（基于公开对比与社区表述，如 [intent-driven.dev](https://intent-driven.dev/knowledge/spec-kit-vs-openspec/)、[OpenSpec 对比页](https://openspec.pro/comparison/) 等）如下：

- **定位**：OpenSpec 为 brownfield/已有代码库设计；Spec Kit 在公开资料中多被描述为 greenfield/新项目优先。
- **规范形态**：OpenSpec 采用统一单源真相、变更用 delta 合并，已有项目下「当前系统约定」一目了然；Spec Kit 按 feature 多文件，拼出全局约定成本更高。
- **流程与迭代**：OpenSpec 三步（Propose → Apply → Archive）、产出更精简；Spec Kit 阶段多、产出更详，已有项目常做小步修改时，轻量流程更匹配。
- **依赖与上手**：OpenSpec 仅需 Node.js、约 5 分钟上手；Spec Kit 需 Python/uv、约 30 分钟，对纯前端或 Node 栈更友好。

以上依据公开对比与社区表述，具体以各工具官方文档与第三方对比文章为准。

### 老项目起步建议

1. **小步试点**：选一个小功能（如加一个 API、修一个 Bug），用 OpenSpec 或 Cursor Plan 模式试一次
2. **逐步推广**：习惯后再用于重构、新模块开发

---

## 六、总结

| 要点 | 说明 |
|------|------|
| **本质** | 先定规范，再写代码；规范即需求、约束和文档 |
| **解决的问题** | 需求漂移、文档滞后、AI 随意发挥、重构恐惧 |
| **Cursor Plan** | 是规范式编程的一种实现，先计划后执行 |
| **老项目** | 推荐 OpenSpec 或 Cursor Plan，从小功能试点开始；已有项目二选一更推荐 OpenSpec（原因见第五节） |

规范式编程不是银弹，但能显著提升可控性和可维护性。在 AI 时代，**把规范写清楚，就是给 AI 一份「施工图」**——图越清晰，实现越靠谱。
