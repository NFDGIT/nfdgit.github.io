Skill 是什么？

概念 类比

想象你请了一个新助手。这个助手很聪明，但什么都不会——你得教他。

你可以给他一个工具（比如一把扳手），他就能拧螺丝。但如果你想让他完成"汽车保养"这样的复杂任务，光给工具不够——你还需要给他一本操作手册，告诉他第一步做什么、第二步做什么、遇到问题怎么处理。

🔧
Tool = 工具
一把扳手，拧一次螺丝
vs
📋
Skill = 操作手册
一套完整的操作流程

在 AI Agent 的世界里，Skill 就是一份写给 Agent 的操作手册。它是一个文本文件（叫 SKILL.md），用人话写清楚"遇到这类任务该怎么做"。Agent 读了这份手册，就学会了一项新技能。

更棒的是：这个手册格式是通用的——写一次，Cursor、Claude Code、Codex 等 16 种以上的 AI 工具都能读懂。

文件结构

text 
my-skill/
  └── SKILL.md          ← 只需要这一个文件
  └── scripts/           ← 可选：辅助脚本
        └── helper.py

Tool vs Skill 技术对比

  • Tool：get_weather(city) — 一次函数调用返回结果，模型自主决定何时调用。
  • Skill：code-review — 一整套工作流（读代码 → 分析问题 → 写报告），按指令一步步执行。
  • Skill 可以内部使用多个 Tool，是更高层的能力封装。

支持 Skill 的工具

  • Cursor、Claude Code、OpenAI Codex、Gemini CLI、VS Code Copilot、Windsurf 等 16+ 工具
  • 都遵循同一个 SKILL.md 标准格式
  • 一个 Skill 写一次，在所有支持的工具中通用

编写你的第一个 Skill

SKILL.md 说明书

写 Skill 就像写一份说明书，只有两部分：

🏷️
封面信息
这个技能叫什么、干什么
+
📝
操作步骤
第一步做什么、第二步做什么

举个真实例子：假设你每次发布项目前都要检查一堆东西（代码有没有提交？测试通过了吗？），你可以写一个"部署检查"技能，以后只要说一句"帮我检查能不能部署"，Agent 就会自动完成所有检查。

封面信息写的是：技能名叫"deploy-checker"，用途是"部署前检查项目状态"。

操作步骤写的是：先检查代码状态，再跑测试，再查依赖，再验证构建，最后用清单汇报结果。

就这么简单——用你自己的话写清楚"遇到这种任务该怎么做"就行。

SKILL.md 由 YAML 头部（封面信息）+ Markdown 正文（操作步骤）组成：

markdown 
---
name: deploy-checker
description: 部署前检查项目状态，确保没有遗漏
license: MIT
compatibility: Cursor, Claude Code
metadata:
  author: your-name
  version: "1.0"
---

# 部署前检查

当用户说"帮我检查一下能不能部署"时，执行以下步骤：

1. **检查 Git 状态**：运行 `git status`，确认没有未提交的更改
2. **运行测试**：执行项目的测试命令，确认全部通过
3. **检查依赖**：确认 package.json / requirements.txt 没有安全漏洞
4. **构建验证**：运行 build 命令，确认没有编译错误

## 输出格式

用清单形式展示每项检查结果：
- ✅ Git 状态干净
- ✅ 测试全部通过（42/42）
- ⚠️ 发现 1 个低风险依赖漏洞
- ✅ 构建成功

最后给出建议：可以部署 / 建议修复后再部署。

YAML 头部字段说明

  • name（必填）：技能名称，用英文短横线连接
  • description（必填）：一句话描述用途，Agent 靠它判断何时启用这个技能
  • license：开源协议
  • compatibility：兼容哪些 AI 工具
  • metadata：作者、版本等附加信息
  • 步骤化：用 1、2、3 编号，让 Agent 按顺序执行
  • 具体化：别说"检查一下"，说"运行 git status 命令，看输出是否包含 'nothing to commit'"
  • 定义输出格式：告诉 Agent 用什么格式汇报结果（清单、表格、摘要）
  • 处理异常：说明出错了怎么办（跳过？换一种方式？告知用户？）

把 Skill 交给 Agent

安装 使用

写好操作手册之后，你需要把它放进 Agent 的工具箱。就像把新买的说明书放到书架上——Agent 会自动翻阅，需要时按手册操作。

在 Cursor 中（最常见的方式）：

  • 把你写的 SKILL.md 放到项目里的 .cursor/skills/你的技能名/ 文件夹下
  • Agent 会自动发现它——你只需要在聊天中正常提需求
  • 比如放了 deploy-checker 技能后，你说"帮我检查能不能部署"，Agent 就会按手册执行
  • 也可以用 @deploy-checker 手动指定使用某个技能

两种放法：

  • 放在项目里（`.cursor/skills/`）：只有这个项目能用，适合项目专属的技能
  • 放在全局目录（`~/.cursor/skills/`）：所有项目都能用，适合通用技能
就是这么简单：写一个文件，放到指定目录，Agent 就学会了新技能。不需要安装、不需要配置、不需要重启。
bash 
# 项目级（仅当前项目可用）
mkdir -p .cursor/skills/deploy-checker
cp SKILL.md .cursor/skills/deploy-checker/

# 全局级（所有项目可用）
mkdir -p ~/.cursor/skills/deploy-checker
cp SKILL.md ~/.cursor/skills/deploy-checker/

不同工具的路径

  • Cursor：.cursor/skills/
  • Claude Code：.claude/skills/
  • OpenAI Codex：.codex/skills/
  • 项目级优先于全局级，同名技能项目级覆盖全局

如果你自己写了 Agent（参考主教程），可以用代码读取 SKILL.md 并注入系统提示：

python 
from pathlib import Path
import yaml

def load_skill(skill_dir):
    """读取 SKILL.md，解析封面信息和操作指令"""
    content = Path(skill_dir, "SKILL.md").read_text(encoding="utf-8")
    parts = content.split("---", 2)
    meta = yaml.safe_load(parts[1])
    instructions = parts[2].strip()
    return {"meta": meta, "instructions": instructions}

def run_agent_with_skill(user_input, skill_dir):
    """加载 Skill 并注入 Agent"""
    skill = load_skill(skill_dir)
    messages = [
        {"role": "system", "content": (
            f"你是一个智能助手。\n\n"
            f"## 可用技能: {skill['meta']['name']}\n"
            f"{skill['meta']['description']}\n\n"
            f"当用户请求与此技能相关的任务时，按以下指令执行：\n\n"
            f"{skill['instructions']}"
        )},
        {"role": "user", "content": user_input}
    ]
    response = client.chat.completions.create(
        model="gpt-4o-mini", messages=messages
    )
    return response.choices[0].message.content

实战：代码审查 Skill

真实案例 代码审查

用一个真实场景来串联前面所学。

场景：你的团队每次改完代码都需要有人审查——检查逻辑对不对、有没有安全问题、代码好不好读。这个过程很重要但很耗时。你可以写一个"代码审查"技能，让 Agent 像一个资深程序员一样帮你审查。

这个 Skill 教 Agent 做什么：

  • 先理解这次代码改了什么、为什么改
  • 从 6 个维度逐一检查：逻辑正确性、安全风险、代码可读性、性能问题、测试覆盖
  • 用红黄绿三色清单汇报问题（严重/建议/良好）
  • 最后给出"通过"或"需要修改"的结论

写好这个技能文件，放到 .cursor/skills/code-review/ 目录下，以后你只要说"帮我审查一下这段代码"，Agent 就会按这套标准自动审查。

核心收获：Skill 本质就是一份标准化的操作指南。写一次，反复用。你的每一份经验和流程都可以变成 Skill，让 Agent 永远按你的标准执行。
markdown 
---
name: code-review
description: 对代码变更进行结构化审查，检查质量、安全和可维护性
license: MIT
metadata:
  author: your-name
  version: "1.0"
  tags: [code-quality, review]
---

# 代码审查技能

收到代码审查请求后，按以下维度逐一检查：

## 审查流程

1. **理解变更**：先读 diff 或文件，理解这次改动的目的
2. **正确性**：逻辑是否正确？边界条件是否处理？
3. **安全性**：有无注入、泄露、硬编码密钥等风险？
4. **可读性**：命名是否清晰？复杂逻辑有无注释？
5. **性能**：有无明显的性能问题（N+1 查询、不必要的循环）？
6. **测试**：变更是否有对应的测试覆盖？

## 输出格式

```
### 审查结果

**变更概述**：一句话描述这次改动

**问题清单**：
- 🔴 [严重] 问题描述 → 建议修复方式
- 🟡 [建议] 问题描述 → 改进方向
- 🟢 [良好] 值得肯定的做法

**总体评价**：通过 / 需要修改后再审
```

## 注意事项

- 只指出真正的问题，不要吹毛求疵
- 给出具体修复建议，不要只说"不好"
- 肯定好的做法，审查不只是找问题
python 
import glob

def load_all_skills(skills_dir=".cursor/skills"):
    """加载目录下所有 Skill"""
    skills = {}
    for skill_file in glob.glob(f"{skills_dir}/*/SKILL.md"):
        skill = load_skill(Path(skill_file).parent)
        skills[skill["meta"]["name"]] = skill
    return skills

def match_skill(user_input, skills):
    """根据用户输入匹配最相关的 Skill"""
    for name, skill in skills.items():
        desc = skill["meta"]["description"].lower()
        if any(kw in user_input.lower() for kw in desc.split()):
            return skill
    return None

# 使用示例
skills = load_all_skills()
skill = match_skill("帮我审查这段代码", skills)
if skill:
    result = run_agent_with_skill("帮我审查这段代码:\n...", skill)
    print(result)

给操作手册附带小工具

脚本 自动化

回到我们的类比：操作手册告诉 Agent "做什么"。但有些步骤比较复杂或重复（比如"统计项目里有多少行代码"），用人话描述很麻烦。这时候你可以给手册附带一些小工具（脚本），让 Agent 直接运行。

就像宜家家具说明书里附带的六角扳手——说明书告诉你"用附带的扳手拧紧螺丝"，你不需要自己去找工具。

具体做法：

  • 在 Skill 目录下建一个 scripts/ 文件夹，放进你写好的脚本
  • 在操作手册（SKILL.md）里写清楚"运行这个脚本"
  • Agent 读到这条指令后会自动执行脚本，拿到结果后继续下一步

如果脚本跑不起来怎么办？

  • 缺少工具（比如没装 Python）：手册里可以写"如果没有 Python，就用其他方式代替"
  • 脚本报错：手册里可以写"如果失败，告诉用户原因，并用手动方式完成"
  • 权限不够：Agent 会请求用户授权，或者用其他方式绕过
总结：操作手册是大脑（告诉 Agent 做什么），脚本是双手（自动完成重复操作）。手册负责编排流程，脚本负责干活。即使脚本有问题，手册里写好了降级方案，Agent 也不会卡住。

目录结构：

text 
project-stats/
  ├── SKILL.md              ← 指令文件
  └── scripts/
        ├── count_lines.py   ← 统计代码行数
        └── check_deps.sh    ← 检查依赖状态

Python 脚本示例（统计代码行数）：

python 
#!/usr/bin/env python3
"""scripts/count_lines.py — 统计项目代码行数"""
import sys, os
from pathlib import Path

def count_lines(directory, extensions=None):
    if extensions is None:
        extensions = ['.py', '.js', '.ts', '.html', '.css']
    stats = {}
    for ext in extensions:
        files = list(Path(directory).rglob(f'*{ext}'))
        lines = sum(len(f.read_text(errors='ignore').splitlines()) for f in files)
        if lines > 0:
            stats[ext] = {'files': len(files), 'lines': lines}
    return stats

if __name__ == '__main__':
    import json
    target = sys.argv[1] if len(sys.argv) > 1 else '.'
    result = count_lines(target)
    print(json.dumps(result, indent=2, ensure_ascii=False))

编写辅助脚本（Bash 示例）：

bash 
#!/bin/bash
# scripts/check_deps.sh — 检查项目依赖状态

set -euo pipefail

echo '{'

# 检查 Node.js
if command -v node &> /dev/null; then
    echo "  \"node\": \"$(node -v)\","
else
    echo '  "node": "NOT_INSTALLED",'
fi

# 检查 Python
if command -v python3 &> /dev/null; then
    echo "  \"python\": \"$(python3 --version 2>&1)\","
else
    echo '  "python": "NOT_INSTALLED",'
fi

# 检查 package.json
if [ -f "package.json" ]; then
    outdated=$(npm outdated --json 2>/dev/null || echo '{}')
    count=$(echo "$outdated" | python3 -c "import sys,json; print(len(json.load(sys.stdin)))" 2>/dev/null || echo "0")
    echo "  \"npm_outdated\": $count,"
else
    echo '  "npm_outdated": "NO_PACKAGE_JSON",'
fi

# 检查 git 状态
if command -v git &> /dev/null && git rev-parse --is-inside-work-tree &> /dev/null; then
    dirty=$(git status --porcelain | wc -l | tr -d ' ')
    echo "  \"git_dirty_files\": $dirty"
else
    echo '  "git_dirty_files": "NOT_A_REPO"'
fi

echo '}'

在 SKILL.md 中引用脚本——关键是在指令中明确告诉 Agent 该运行什么命令：

markdown 
---
name: project-stats

# 项目统计技能

当用户要求分析项目状态时，按以下步骤执行：

## 前置检查（每次执行前必须完成）

先检查所需工具是否已安装：
```bash
which python3 && which git && which npm
```

- 如果 `python3` 不存在 → 告知用户："需要安装 Python 3 才能运行统计脚本。请运行 `brew install python3`（macOS）或 `apt install python3`（Linux）"，然后**跳过**代码行数统计步骤，用 `find` + `wc -l` 命令手动统计作为降级方案。
- 如果 `git` 不存在 → 跳过 Git 状态检查，说明原因。
- 如果 `npm` 不存在 → 跳过依赖检查，说明原因。

## 步骤

1. **统计代码行数**
   运行脚本获取各语言的文件数和行数：
   ```bash
   python3 .cursor/skills/project-stats/scripts/count_lines.py .
   ```
   解析 JSON 输出，整理成表格展示。

   **如果脚本执行失败**（非零退出码或报错）：
   - 读取 stderr 输出，告知用户具体错误
   - 降级方案：用 shell 命令手动统计
     ```bash
     find . -name "*.py" -o -name "*.js" -o -name "*.ts" | head -100 | xargs wc -l
     ```

2. **检查依赖状态**
   运行依赖检查脚本：
   ```bash
   bash .cursor/skills/project-stats/scripts/check_deps.sh
   ```

   **如果脚本返回 NOT_INSTALLED**：
   - 对应工具未安装，在报告中标注 ⚠️ 并给出安装命令
   - 不要因为部分检查失败就中断整个流程

   **如果脚本本身无法执行**（如权限不足）：
   - 尝试 `chmod +x` 后重试
   - 仍然失败则手动执行等效命令：
     ```bash
     node -v 2>/dev/null || echo "Node.js 未安装"
     python3 --version 2>/dev/null || echo "Python 未安装"
     git status --porcelain 2>/dev/null | wc -l
     ```

3. **生成报告**
   将以上信息整合为 Markdown 格式的项目概况报告。
   如果某些步骤被跳过或降级，在报告末尾注明原因。

## 注意

- 脚本路径使用完整的 Skill 目录路径
- **永远不要因为一个脚本失败就放弃整个任务**——降级、跳过、手动替代
- 不要修改项目文件，这是只读分析
- 所有 shell 命令失败时读取 stderr 并告知用户

Agent 怎么执行脚本？

  • 在 Cursor / Claude Code 中：Agent 通过内置的 Shell 工具执行命令。你在 SKILL.md 中写的 bash 代码块，Agent 会识别为需要运行的命令。
  • 在自写 Agent 中：你需要给 Agent 一个 run_command 工具（用 subprocess 实现），让它能执行 shell 命令。

脚本路径技巧

  • 使用完整路径引用脚本：.cursor/skills/<skill-name>/scripts/xxx.py
  • 或在 SKILL.md 中指示 Agent 先 cd 到 Skill 目录再运行
  • 脚本接收参数用 sys.argv（Python）或 $1 $2（Bash）
  • 脚本输出用 JSON 格式，方便 Agent 解析结果

安全注意事项

  • Cursor 的沙箱会限制某些操作（网络、USB 等），脚本需要额外权限时 Agent 会请求用户授权
  • 给脚本加可执行权限：chmod +x scripts/*.sh
  • 避免在脚本中硬编码敏感信息，用环境变量传入
  • 安装社区 Skill 前务必审查脚本内容
