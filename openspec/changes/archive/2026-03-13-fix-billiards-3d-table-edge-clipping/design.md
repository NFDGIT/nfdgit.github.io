## Context

当前台球页 3D 渲染已接入 Three.js，但在部分视口下存在“可碰撞边界与可视边界不一致”：球桌边缘被裁切、边框显示不全、球接近边缘时离开可见区域。该问题在切换 2D/3D、窗口 resize、移动端横竖屏切换后更明显，说明相机参数、渲染尺寸同步与容器裁剪策略之间存在耦合缺口。

## Goals / Non-Goals

**Goals:**

- 让 3D 模式下球桌完整可见（桌面、边框、袋口不被裁切）。
- 让球在 2D 物理边界附近仍保持可见，避免“球还在桌上但画面消失”。
- 保证在视图切换、窗口缩放、移动端横竖屏切换后，3D 视口稳定且不丢边缘内容。

**Non-Goals:**

- 不改变台球物理规则（碰撞、摩擦、进袋、计分）。
- 不新增渲染库或构建流程。
- 不重做 UI 布局，仅在必要时微调 3D 容器样式。

## Decisions

- **统一以逻辑桌面边界驱动 3D 相机 framing**：相机位置、FOV 与 lookAt 基于 `LOGIC_W/LOGIC_H` 和桌边安全边距计算，确保桌面外框在目标宽高比下完整进入视锥，而不是依赖固定经验值。  
  - 备选方案：仅增大相机高度或缩小 FOV。  
  - 取舍：简单但不稳，在不同纵横比下仍可能裁切，故不采用。

- **强化渲染尺寸同步时机**：在 `setRenderMode('3d')`、`resize`、`orientationchange`（或等效 resize 流程）中统一调用同一套 3D viewport 同步逻辑，保证 renderer size、camera aspect、projection matrix 一致更新。  
  - 备选方案：仅依赖现有 `resize` 监听。  
  - 取舍：切换视图后的首帧常出现尺寸未就绪问题，需在模式切换后补一次同步。

- **显式约束 3D 容器裁剪行为**：确保 3D 容器与 canvas 显示边界一致，不出现额外的 overflow 裁掉桌边。  
  - 备选方案：不改 CSS，只调相机。  
  - 取舍：如果容器链路本身裁剪，单改相机无法彻底解决，需双向校正。

- **以“逻辑边界可见性”作为验收基准**：验证点不只看静态首帧，还要覆盖“球到边缘、切视图、改尺寸”三类动态过程。

## Risks / Trade-offs

- **[风险]** 调整相机 framing 可能改变现有 3D 视觉透视感。  
  **缓解：** 在“完整可见”优先前提下，保留适度倾斜视角并限定参数范围，避免风格突变。

- **[风险]** 频繁 resize 同步可能引入额外开销。  
  **缓解：** 复用已有渲染循环，仅在必要事件触发时更新 camera/renderer。

- **[风险]** 2D/3D 显示边界对齐后暴露既有样式问题。  
  **缓解：** 将 CSS 改动限定在 3D 容器相关选择器，避免波及其他页面模块。

## Migration Plan

- 无数据迁移。
- 实施后执行回归：桌面端常见分辨率、移动端竖屏/横屏、2D/3D切换、拖拽击球到边缘、窗口 resize。
- 若异常，可回滚到本次变更前的 `billiards.js` 3D 相机与尺寸同步逻辑。
