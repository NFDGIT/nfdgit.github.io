## Context

站点已有赛车和贪吃蛇两款 Canvas 游戏，均以独立子目录（`games/racing/`、`games/snake/`）的方式组织。台球游戏在物理仿真上比这两款更复杂，需要处理圆形碰撞、摩擦衰减和多球交互，但依然适合用纯 Canvas 在浏览器中实现。本次以二维俯视视角为基础，不引入 3D 渲染引擎，控制实现复杂度。

## Goals / Non-Goals

**Goals:**
- 在 `games/billiards/` 下提供一个独立的二维台球游戏页面
- 用 Canvas 实现台球桌、球（母球 + 彩球）、球袋和瞄准线的渲染
- 实现基础的物理引擎：球的速度与位移、球与球的碰撞、球与桌边的弹射、进袋判定、摩擦衰减
- 支持鼠标（拖拽或点击）进行瞄准和击球，并在移动端支持触控操作
- 提供重置、积分和通关（全部彩球进袋）判定
- 把台球游戏入口加入目录页
- 保持纯静态、零外部依赖、兼容站点主题变量

**Non-Goals:**
- 不实现真实的台球规则（开球犯规、8 球规则、花式台球等）
- 不引入 3D、WebGL 或外部物理引擎库
- 不实现多人联机或 AI 对手
- 不处理复杂的音效或粒子特效

## Decisions

1. **渲染方案：单层 Canvas**
   - 用一个 `<canvas>` 元素渲染全部内容（桌面、球、瞄准线、UI）。
   - 备选：多层 Canvas（背景层 + 游戏层）。
   - 未采用：当前球数量有限（16 以内），单层 Canvas 逐帧全量重绘的性能足够，多层会增加实现复杂度。

2. **物理模型：简化圆形碰撞 + 摩擦衰减**
   - 每帧按速度向量更新球的位置，碰撞时用弹性碰撞公式更新速度，每帧乘以衰减系数模拟摩擦。
   - 备选：引入 Matter.js 等物理引擎。
   - 未采用：引入外部依赖与项目零依赖约定冲突，且对二维台球来说手写物理可控性更好。

3. **瞄准与击球输入：拖拽方向决定击球角度和力度**
   - 用户从母球位置向外拖拽，线段长度映射击球力度，方向为击球方向；松开后发球。
   - 备选：固定强度 + 角度拨盘。
   - 未采用：拖拽方式在触控和鼠标上都更直观，不需要额外 UI 控件。

4. **游戏目录入口：更新 `games/index.html`**
   - 在目录页新增一张台球游戏卡片，路径指向 `games/billiards/index.html`。
   - 与现有赛车和贪吃蛇卡片保持一致的结构和样式。

5. **移动端适配：Canvas 等比缩放 + 触控事件**
   - Canvas 尺寸根据可用视口宽度等比缩放，绘制坐标统一以逻辑尺寸计算。
   - 补充触控事件（`touchstart`、`touchmove`、`touchend`）与鼠标事件统一处理。

## Risks / Trade-offs

- **[风险]** 简化物理引擎可能产生"穿球"或碰撞抖动
  - **缓解**：每帧后对重叠球做位置分离修正；若仍有问题可降低每帧移动步长

- **[风险]** Canvas 在移动端缩放后触控坐标需要转换，容易出现偏移
  - **缓解**：统一使用 `getBoundingClientRect()` 转换坐标，不依赖 `offsetX/Y`

- **[风险]** 游戏逻辑较复杂，单文件 JS 可能难以维护
  - **缓解**：在同一个 IIFE 内按职责分区（物理、渲染、输入、状态），不强制模块化
