## Context

台球游戏位于 `games/billiards/`，当前使用单一 Canvas 2D 渲染：球桌、球、瞄准线、力度条均在 2D 坐标系下绘制；物理（碰撞、摩擦、进袋）与状态（积分、重置、通关）均在现有 `billiards.js` 中实现。站点为纯静态、无构建，依赖仅通过 script 标签引入。本次变更为该游戏增加 3D 显示与 2D/3D 切换，不改变玩法与物理规则。

## Goals / Non-Goals

**Goals:**

- 增加 3D 渲染模式：使用 WebGL 渲染三维球桌、球体与球袋，相机俯视或适度倾斜，光照与材质体现立体感。
- 提供 2D/3D 模式切换入口，两种模式下玩法、物理、积分与通关逻辑一致；瞄准、力度条、触控/鼠标操作在 3D 模式下同样可用。
- 保持无构建：通过 CDN 引入 3D 库（如 Three.js），不新增打包步骤。
- 保持主题与移动端适配：3D 场景颜色或光照与站点主题协调；窄屏下 3D 可正常显示与操作。

**Non-Goals:**

- 不实现真实 3D 物理引擎（仍用现有 2D 物理，仅用 3D 做视觉呈现）。
- 不做自由相机漫游或复杂相机控制，仅固定或简单可调视角即可。
- 不移除 2D 模式，不破坏现有 2D 体验。

## Decisions

- **3D 库选型**：采用 **Three.js**（通过 CDN，如 unpkg 或 jsDelivr），理由：生态成熟、文档多、纯前端即可、无需构建；替代方案如原生 WebGL 或 Babylon.js 会增加实现量或体积，故不选。
- **渲染与逻辑分离**：物理与游戏状态仍由现有 2D 逻辑驱动（球的位置、速度、进袋、积分等）；3D 层每帧根据当前状态更新场景中球与瞄准线的位置/方向，不重复实现碰撞与规则。即「逻辑 2D、渲染可选 2D 或 3D」。
- **2D/3D 切换方式**：在页面增加切换控件（如按钮或下拉），切换时：若选 3D，隐藏或停用 2D Canvas，在相同容器或另一容器中挂载 Three.js 的 renderer.domElement，并复用同一套输入与状态；若选 2D，销毁或隐藏 3D 场景，恢复 2D Canvas 渲染。同一时刻仅一种渲染激活。
- **相机与视角**：3D 模式默认正交或透视相机俯视（可带小倾角），相机位置固定或提供 1～2 档预设；不在首版做拖拽旋转相机，以降低复杂度和移动端误触。
- **降级策略**：若 WebGL 不可用或 Three.js 加载失败，保持或回退到 2D 模式，并可选提示「3D 不可用」。

## Risks / Trade-offs

| 风险 | 缓解 |
|------|------|
| 移动端 3D 性能不足 | 3D 模式下控制 draw call 与面数（简单几何体、少量光）；可选在窄屏默认 2D 或提供「低画质」选项。 |
| 两套渲染代码维护成本 | 抽离「游戏状态 → 视图」的接口，2D/3D 共用一个状态源，避免双份逻辑。 |
| CDN 依赖与加载失败 | 使用可靠 CDN、fallback 到 2D；必要时可后续改为自托管 Three.js。 |

## Migration Plan

- 无数据迁移。上线时先部署到 `games/billiards/`，保留 2D 为默认或与 3D 并列可选；若 3D 有严重问题可仅隐藏切换入口，2D 照常使用。
- 回滚：移除 3D 相关脚本与切换 UI 即可恢复为纯 2D 版本。

## Open Questions

- 无。实现时若移动端 3D 卡顿，再考虑「窄屏默认 2D」或简化 3D 场景。
