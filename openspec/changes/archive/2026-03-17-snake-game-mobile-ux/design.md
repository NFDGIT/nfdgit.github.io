## Context

贪吃蛇游戏当前已可在移动端运行：页面响应式可用，并在侧栏提供了方向按钮（上/下/左/右 + 暂停）。然而实际体验差距明显——方向按钮的默认 `min-height: 3.25rem` 在拇指操作下偏小，不支持滑动手势，画布在窄屏下可能被侧栏挤压导致需要滚动才能看到完整区域，游戏进行中触摸画布会触发页面滚动。

技术栈为纯静态 HTML/CSS/JS，无框架依赖。触控逻辑需直接在 `snake.js` 中通过原生 Touch API 实现。

## Goals / Non-Goals

**Goals:**

- 移动端用户可以通过滑动手势控制蛇的方向，体验流畅自然
- 所有触控目标在窄屏下满足 ≥44px 的最小可点击区域
- 移动端布局保证画布优先展示、无需滚动即可操作
- 游戏进行中不会因触摸操作导致页面滚动或浏览器默认手势
- 桌面端行为完全不受影响

**Non-Goals:**

- 不做虚拟摇杆 / 模拟手柄等复杂输入方案
- 不引入第三方手势库（如 Hammer.js）
- 不重构整体游戏架构；保持现有 IIFE + 状态对象的结构
- 不增加新的游戏玩法或模式

## Decisions

### 1. 滑动手势识别方案

**决定**：在 `snake.js` 中直接监听 `touchstart` / `touchend` 事件，通过起终点坐标差计算滑动方向，设定最小滑动距离阈值（30px）以过滤误触。

**理由**：原生 Touch API 足以识别四方向滑动，无需引入 `touchmove` 中间帧追踪或第三方库，代码量少、无新依赖、与现有 `queueDirection()` 逻辑无缝对接。

**备选**：使用 Hammer.js 等手势库——但项目坚持零依赖原则，不引入。

### 2. 手势监听区域

**决定**：将手势监听绑定在 `.snake-canvas-wrap` 元素上（即画布容器），而非全页面或 document。

**理由**：限定在画布区域可以避免在侧栏按钮区触发误判，也避免与页头导航的触摸行为冲突。画布是用户游戏时注视和触摸最多的区域。

**备选**：绑定在 document 上——会与页面导航、按钮区域冲突。

### 3. 防滚动策略

**决定**：在 CSS 中为 `.snake-canvas-wrap` 添加 `touch-action: none`，在 JS 的 touchstart handler 中对画布区域调用 `e.preventDefault()`。

**理由**：`touch-action: none` 告诉浏览器此区域不需要默认触摸行为（滚动、缩放），结合 `preventDefault()` 双重保障，兼容 iOS Safari 和 Chrome。不影响画布区域外的正常滚动。

### 4. 移动端布局调整

**决定**：在 `@media (max-width: 768px)` 中将 `.snake-layout` 改为单列布局，画布区域在上、侧栏在下；画布容器宽度设为 `100%` 并保持正方形比例；侧栏中「玩法说明」折叠或隐藏以减少滚动。

**理由**：移动端首要目标是画布可见可操作。当前 900px 断点切单列，但画布仍是固定 432px 宽度未自适应；改为百分比宽度后画布随屏幕缩放，减少横向溢出。

### 5. 触控按钮热区扩大

**决定**：在 `@media (max-width: 768px)` 中将 `.snake-control` 的 `min-height` 从 3.25rem 提升到 3.5rem，`.snake-button` 的 `min-height` 设为 2.75rem，确保均 ≥44px。

**理由**：Apple HIG 和 WCAG 建议触控目标不小于 44×44px。3.5rem（56px）在大部分基准字号下足够。

### 6. 触觉反馈（可选）

**决定**：在方向切换成功时调用 `navigator.vibrate(10)`，仅当 `navigator.vibrate` 存在时执行，不做 polyfill。

**理由**：微弱震动可增强操作反馈，代码一行，无副作用。iOS Safari 不支持此 API，静默降级。

## Risks / Trade-offs

- **[滑动手势与按钮共存]** → 两套输入同时可用可能让用户困惑。缓解：保留按钮但在移动端通过视觉提示（如 overlay 文案或 subtitle）引导用户尝试滑动。
- **[touch-action: none 范围]** → 仅限于画布容器，若用户在画布区域想滚动页面将无法滚动。缓解：游戏进行中用户预期是在操作游戏而非滚动页面，可接受。
- **[画布缩放后清晰度]** → Canvas 用 CSS 缩放可能模糊。缓解：在 resize 时同步更新 canvas 的 width/height 属性（现有 `resizeCanvas()` 已处理）。
