## Context

当前台球游戏在 2D 与 3D 模式均采用“拖拽方向 = 出杆方向”的交互规则。新需求要求将其改为“拖拽方向与出杆方向相反”，即玩家向某方向拖拽时，母球应朝反方向发射。该变更属于交互语义调整，不涉及物理引擎、球体碰撞、进袋判定、积分与通关逻辑。

## Goals / Non-Goals

**Goals:**

- 将瞄准与出杆方向映射统一改为“反向出杆”。
- 在 2D 与 3D 下保持一致行为，避免模式间方向语义不一致。
- 不改变力度计算区间与反馈方式（仅方向翻转）。

**Non-Goals:**

- 不重写碰撞/摩擦/进袋等物理逻辑。
- 不调整 UI 布局与主题样式。
- 不引入新依赖。

## Decisions

- **方向向量统一反转**：在逻辑层将击球方向从 `aimCurrent - aimStart` 改为 `aimStart - aimCurrent`（或等价地在归一化后乘以 `-1`），确保同一拖拽在所有模式都得到反向出杆。
- **瞄准线与力度语义同步**：瞄准线渲染方向需与实际出杆方向一致，避免“线朝一边、球朝另一边”；力度仍由拖拽距离决定，不因方向变更而改变。
- **2D/3D 共用同一方向计算**：3D 仅负责显示，不单独实现方向规则，继续复用统一逻辑层结果，降低分叉行为风险。
- **优先修改现有 requirement**：在 `billiards-game` 现有 requirement 上做 MODIFIED，避免新增冗余 requirement 导致规范冲突。

## Risks / Trade-offs

| 风险 | 缓解 |
|------|------|
| 玩家已形成旧手感，变更后短期不适应 | 在玩法说明中明确“反向出杆”规则，并保持 2D/3D 一致减少认知成本。 |
| 只改出杆方向未改瞄准线会造成误导 | 同步修改瞄准线方向渲染，并在验收中检查“线与球同向”。 |
| 2D/3D 计算路径出现分叉导致行为不一致 | 方向计算集中在逻辑层，3D 仅消费结果。 |

## Migration Plan

- 无数据迁移。
- 发布前执行回归：2D/3D 下同向拖拽均应反向出杆；力度条与击球力度关系不变。
