## Context

站点为纯静态 HTML/CSS/JS 项目，`games/` 目录下已有独立游戏页面，但整体实现偏散，缺少一个风格更统一、易于维护的轻量小游戏。此次新增贪吃蛇需要保持零构建、零依赖，能直接部署到 GitHub Pages，并兼顾桌面端键盘操作与移动端触控操作。

## Goals / Non-Goals

**Goals:**
- 在 `games/` 目录下新增一个可独立访问的贪吃蛇页面
- 使用 Canvas 或等效轻量渲染方案完成棋盘、蛇身和食物绘制
- 提供开始、暂停、重新开始、分数展示和游戏结束反馈
- 同时支持键盘方向键与触控按钮输入
- 页面样式遵循现有站点的主题变量与响应式约束

**Non-Goals:**
- 不引入 3D、物理引擎或外部游戏框架
- 不增加排行榜、联网对战或存档能力
- 不改造现有其他游戏的实现方式
- 不处理复杂音效、成就系统或多关卡设计

## Decisions

1. **页面组织**：新增独立页面承载贪吃蛇，而不是复用现有赛车游戏页面结构。这样可以避免与现有脚本强耦合，也更符合每个游戏一个入口的目录习惯。
   - 备选方案：将贪吃蛇逻辑嵌入 `games/index.html`
   - 未采用原因：现有 `games/index.html` 已承载另一款游戏资源，直接复用会增加冲突和维护成本。

2. **渲染方案**：采用二维网格渲染，优先使用单个 Canvas 绘制棋盘、蛇和食物。
   - 备选方案：使用 DOM 网格逐格渲染
   - 未采用原因：DOM 网格在频繁刷新下更容易产生样式和性能负担，而 Canvas 更适合持续重绘的小游戏。

3. **状态管理**：游戏逻辑采用简单状态机，至少区分 `idle`、`running`、`paused`、`gameover` 四种状态，并由统一循环驱动移动与碰撞判定。
   - 备选方案：用多个 `setInterval` 分别管理移动、计时和 UI
   - 未采用原因：多计时器更容易造成状态不同步，统一循环更容易暂停、重启和调速。

4. **输入策略**：键盘方向键与屏幕按钮统一映射到同一套转向逻辑，并阻止蛇直接反向移动。
   - 备选方案：桌面端仅支持键盘、移动端依赖浏览器虚拟键盘
   - 未采用原因：移动端体验较差，且无法保证触控设备上有稳定输入方式。

5. **样式与适配**：页面使用 CSS 变量适配浅色/深色主题，布局采用移动端优先设计，在窄屏下将状态区、画布和按钮区垂直堆叠。
   - 备选方案：只为桌面端设计固定尺寸画布
   - 未采用原因：与项目响应式约束不符，也会降低移动端可玩性。

## Risks / Trade-offs

- **[风险]** Canvas 尺寸与设备像素比处理不当会导致画面模糊
  - **缓解**：实现时统一管理逻辑网格尺寸与实际像素尺寸，并优先保证格子边界清晰

- **[风险]** 触控按钮与页面滚动手势冲突，影响移动端操作
  - **缓解**：按钮区使用明确的点击目标和 `touch-action` 控制，避免误滚动

- **[风险]** 游戏速度、网格大小和屏幕尺寸之间存在体验权衡
  - **缓解**：初版固定一组经过验证的默认参数，后续如有需要再扩展难度选项

- **[风险]** 若未提供清晰入口，新增游戏可能难以被用户发现
  - **缓解**：将“补充游戏目录入口”纳入任务清单，作为实现时的显式检查项
