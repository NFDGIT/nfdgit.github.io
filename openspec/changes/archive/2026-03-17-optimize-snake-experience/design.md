## Context

当前贪吃蛇游戏是纯静态 HTML/CSS/JS 实现，使用 Canvas 2D 渲染，18×18 网格，160ms 固定步进。移动端通过 D-pad 按钮控制方向，没有触摸滑动支持。游戏无视觉反馈动画，吃食物只是分数+1，死亡只是弹出 overlay，整体体验缺乏"游戏感"。

## Goals / Non-Goals

**Goals:**
- 移动端可在画布上滑动控制蛇的方向，操控自然直觉
- 吃食物和死亡时有明确的视觉反馈，增强游戏沉浸感
- 随分数递增逐步加快速度，让游戏有挑战递进感
- 蛇身移动平滑不跳跃，视觉质量提升

**Non-Goals:**
- 不添加音效（静音适合该项目场景）
- 不添加联网排行榜
- 不改变核心游戏规则（网格大小、碰撞逻辑等不变）
- 不引入任何外部依赖库

## Decisions

### 决策 1：触摸滑动手势识别方案

**选择**：在 `touchstart`/`touchend` 事件中计算滑动向量，取绝对值较大的轴作为方向。

**理由**：
- 相比 `touchmove` 持续监听，`touchstart` + `touchend` 两点计算更简单高效
- 设置最小滑动距离阈值（30px）避免误触
- 在 `.snake-canvas-wrap` 上设置 `touch-action: none` 阻止浏览器滚动

**替代方案**：
- 使用 Hammer.js 等手势库 → 违反无外部依赖原则
- 监听 `touchmove` 实时计算 → 过度复杂，对贪吃蛇场景无必要

### 决策 2：平滑移动插值方案

**选择**：在渲染函数中根据 `accumulator / STEP_MS` 比率对蛇头位置做线性插值。

**理由**：
- 利用现有 `accumulator` 机制，无需额外计时器
- 仅插值蛇头位置（其余段跟随），计算开销极小
- 插值仅影响渲染，不影响逻辑层的网格对齐碰撞检测

**替代方案**：
- 降低 STEP_MS 提高帧数 → 改变游戏节奏，不是真正的平滑
- CSS transform 动画 → Canvas 渲染无法使用 CSS 动画

### 决策 3：粒子动画系统设计

**选择**：轻量粒子数组，每次吃食物时在食物位置生成 8-12 个粒子，每帧更新位置和透明度，生命周期 400ms 后移除。

**理由**：
- 纯 Canvas 绘制，与现有渲染流程一致
- 粒子数量少、生命周期短，不会影响性能
- 粒子颜色取食物颜色的 CSS 变量，自动适配主题

### 决策 4：难度递进公式

**选择**：`currentStepMs = max(BASE_STEP - score * SPEED_INCREMENT, MIN_STEP)`，即 `max(160 - score * 3, 70)`。

**理由**：
- 线性递减直观易理解，每吃一个食物快 3ms
- 下限 70ms 防止速度过快无法操控
- 仅修改 `STEP_MS` 的运行时值，对现有 accumulator 循环无侵入

**替代方案**：
- 分段式难度（每 10 分提一档）→ 过于突兀，线性更平滑
- 指数递减 → 前期太慢后期太快，体验不佳

### 决策 5：死亡反馈动画

**选择**：画布容器 CSS `shake` 关键帧动画 + Canvas 内蛇身闪烁 2-3 次。

**理由**：
- CSS 震动作用于容器 DOM 元素，不干扰 Canvas 内部渲染
- Canvas 内蛇身闪烁通过交替渲染/跳过实现，逻辑简单
- 动画时长 500ms 内完成，不阻塞 overlay 显示

## Risks / Trade-offs

- **[触摸事件与按钮冲突]** → D-pad 按钮区域不注册滑动，仅画布区域响应手势。两套操控方式并存互不干扰。
- **[插值造成视觉穿墙]** → 插值仅用于渲染，碰撞检测仍在逻辑层的整数网格上判定，不存在真正穿墙。
- **[速度过快影响可玩性]** → 设置 70ms 下限，约 14fps 移动速度，仍在人类反应范围内。
- **[低端设备粒子性能]** → 粒子数最多 12 个且生命周期极短，Canvas 2D 绑定无压力。
