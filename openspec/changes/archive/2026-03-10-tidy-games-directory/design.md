## Context

当前 `games/` 目录已经有正式目录页、赛车页、贪吃蛇页和若干共享脚本，但同时还残留历史 HTML 文件与分散资源，目录职责不够明确。随着后续继续新增台球、桌球等游戏，如果不先做一次结构规整，页面、资源和入口会继续堆在同一层级，维护成本会明显上升。

## Goals / Non-Goals

**Goals:**
- 明确区分游戏目录页、具体游戏页、共享资源和单个游戏私有资源
- 让正式游戏入口只指向整理后的页面，不暴露遗留测试或历史文件
- 为后续继续新增游戏提供可复用的目录组织模式
- 在规整路径和命名的同时，保持现有正式游戏可访问
- 不引入构建工具，继续保持纯静态站点结构

**Non-Goals:**
- 不新增新的游戏玩法或大幅重写现有游戏逻辑
- 不将整个项目都迁移为模块化打包工程
- 不对非 `games/` 模块的目录结构做同等范围的整理
- 不要求本次解决所有第三方源码风格不一致的问题

## Decisions

1. **目录分层**：将 `games/` 内的资源按职责分层，例如“目录页/具体页面/共享脚本/私有资源”几个层级，而不是继续全部平铺在同一层。
   - 备选方案：保留当前平铺结构，仅修补命名
   - 未采用原因：只改命名无法解决后续继续扩展时的拥挤问题。

2. **正式入口收敛**：目录页只展示整理后的正式游戏页面，历史遗留 HTML 文件不再作为用户可见入口。
   - 备选方案：保留遗留页面并继续暴露在目录中
   - 未采用原因：会让目录页再次变杂，也不利于用户理解哪些是正式内容。

3. **共享资源优先复用**：多个游戏共用的第三方脚本、样式或资源优先放入共享位置，单个游戏专有资源再放入各自子目录。
   - 备选方案：每个游戏各自复制一份共享资源
   - 未采用原因：会增加重复文件和后续同步修改成本。

4. **迁移时保持可访问**：页面和资源迁移时，优先通过同步更新引用路径来保持现有正式页面可访问，而不是先删后补。
   - 备选方案：先大规模移动，再统一修错
   - 未采用原因：更容易导致临时断链，且排查困难。

## Risks / Trade-offs

- **[风险]** 文件移动后容易出现脚本、图片或样式引用失效
  - **缓解**：把“路径校验”作为独立任务，对目录页和正式游戏页逐个验证

- **[风险]** 历史遗留文件是否仍有价值存在不确定性
  - **缓解**：优先将其移出正式入口范围，必要时保留但明确归档位置，而不是直接混在主目录中

- **[风险]** 共享资源抽离后，具体游戏页面的相对路径会更复杂
  - **缓解**：统一目录约定，并在整理时一起更新页面引用，避免局部例外
