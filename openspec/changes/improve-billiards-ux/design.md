## Context

台球游戏已可在桌面端运行，但移动端体验存在明显问题：
1. 当前瞄准机制要求点中母球（触发半径约 33 逻辑像素），在竖屏手机上母球缩放后非常小，精准点击困难
2. 固定逻辑分辨率 760×420 在窄屏下导致桌面过小，球很难拖拽操作
3. 侧边面板被推到画布下方，操作按钮不在手指自然到达范围
4. 无力度视觉反馈，拖拽长短不易感知

本次改动的目标是让手机端游戏可以正常进行，同时不破坏桌面端现有体验。

## Goals / Non-Goals

**Goals:**
- 将瞄准触发逻辑改为在 Canvas 上任意位置拖拽，方向指向母球即为击球方向
- 新增力度进度条，在画布下方实时反映当前拖拽力度
- 移动端底部固定控制条（状态 + 按钮），让游戏区尽量大
- Canvas 逻辑尺寸在窄屏下保持更大的桌面宽高比，球的逻辑半径随 Canvas 宽度等比放大
- 改进瞄准线为带箭头的实线，在母球上显示方向圆环
- 保持桌面端现有布局不变

**Non-Goals:**
- 不改动物理引擎核心逻辑（碰撞、衰减）
- 不引入横屏强制要求
- 不改变游戏规则本身

## Decisions

1. **任意位置拖拽触发瞄准**
   - 原方案：只有在母球区域内 `pointerdown` 才触发
   - 新方案：在 Canvas 上任意 `pointerdown` 都触发；将拖拽起点与母球之间的连线方向作为击球反方向；拖拽距离（从任意起点到当前点）映射力度
   - 理由：消除"点不中母球"这一最大痛点，操作更接近触屏直觉

2. **力度进度条**
   - 在画布正下方新增一个细长进度条 `<div>`，`width` 随拖拽距离实时更新（0%–100%），颜色从绿渐变到红
   - 备选方案：Canvas 内绘制力度弧形
   - 未采用原因：DOM 进度条更易控制样式、不影响 Canvas 帧率

3. **移动端底部固定控制条**
   - 在 `<= 700px` 时，将状态信息与操作按钮提取到固定底部条（`position: fixed; bottom: 0`），Canvas 区域 `padding-bottom` 留出空间
   - 备选方案：仅保留浮动的重置按钮
   - 未采用原因：状态信息（已进袋/剩余球）对玩家有用，放底部固定条可以在玩球时随时看到

4. **逻辑桌面尺寸自适应**
   - 桌面端保持原有 760×420
   - 在 Canvas 宽度 `< 500px`（竖屏手机）时将逻辑尺寸调整为 420×280，球的逻辑半径从 11 升至 14
   - 让球在手机上相对更大，拖拽瞄准更准确
   - 重新计算球的初始排列位置以适配新尺寸

5. **瞄准线改进**
   - 原方案：虚线 + 黄色拖拽线
   - 新方案：白色方向实线（从母球延伸方向，带小箭头），橙色力度线（从拖拽起点到鼠标），母球上叠加一圈半透明方向指示圆环
   - 理由：箭头更直观传达"球要往哪里走"

## Risks / Trade-offs

- **[风险]** 任意位置拖拽可能与滚动手势冲突，导致页面被意外滚动
  - **缓解**：Canvas 上已有 `touch-action: none`，并在 `touchmove` 里 `preventDefault()`；底部固定控制条不在 Canvas 上，不影响 Canvas 事件

- **[风险]** 移动端切换为更小逻辑尺寸后，彩球初始三角形排列需要重新计算，可能碰到桌边
  - **缓解**：在 `createBalls` 里用 TABLE_W/TABLE_H 计算相对位置，不使用硬编码坐标

- **[风险]** 底部固定控制条遮住部分画布
  - **缓解**：在 `.billiards-canvas-wrap` 加 `padding-bottom` 补偿固定条高度
