## Context

`note/Flutter/` 已具备底层原理、面试（原理/实践）、高阶用法及 README 导航，且满足 flutter-note-doc 的体量与文风要求。本次变更为「继续补充」：在不破坏现有结构的前提下增加主题覆盖与考点深度。

## Goals / Non-Goals

**Goals:**

- 在现有三块（底层原理、面试、高阶用法）中增加若干新小节、考点或例题，或新增 1～3 个独立主题文档（如工程实践、调试与排错、包与插件），并在 README 中提供入口。
- 补充内容保持生动简练、可参考官方/社区资料；新增或修改后运行 manifest 生成脚本，保证笔记浏览器能正确展示。

**Non-Goals:**

- 不强制规定本次补充的精确字数或倍数；不重写已有章节结构；不引入非静态资源或构建流程。

## Decisions

1. **补充形式**  
   优先在现有文件中增加小节（如底层原理加「常见坑与排查」、面试加若干新题、高阶用法加「发布与 CI 要点」）；若单文件过长或主题独立性强，可新增独立 md（如 `工程实践.md`、`调试与排错.md`），在 README 表格中增加一行导航。具体篇目与标题由 tasks 列出，实现时按 tasks 执行。

2. **资料来源与文风**  
   与现有约定一致：参考官方文档、Flutter 中文网、高星仓库或高赞总结，择优整理；用语简练、可带「一句话」小结，避免大段复制。

3. **Manifest**  
   新增或重命名 md 后必须执行 `node scripts/generate-note-manifest.js`，以便笔记浏览器侧栏与内嵌内容更新。

## Risks / Trade-offs

- **Risk**：补充过多导致单文件过长，影响查阅体验。  
  **Mitigation**：以「小节/题目」为主；若某主题明显独立则拆成新文件，README 做好分区说明。
- **Trade-off**：不在此变更中强制「再扩 5 倍」等数字目标，仅约定「有实质补充且保持质量」，便于后续按需迭代。
